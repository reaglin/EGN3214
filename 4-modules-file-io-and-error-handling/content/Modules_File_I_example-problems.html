What Is an Exception? – Identifying Runtime Errors

  • **Problem:** A student writes the following script to divide two numbers entered by the user. Identify which line will raise an exception, name the exception type, and explain why it occurs.
  • ```
  • x = int(input("Enter numerator: "))
  • y = int(input("Enter denominator: "))
  • result = x / y
  • print("Result:", result)
  • ```
  • The user enters 10 for x and 0 for y.
  • **Step 1 – Trace execution line by line.**
  • Line 1 executes successfully: x = 10.
  • Line 2 executes successfully: y = 0.
  • Line 3 attempts to evaluate 10 / 0.
  • **Step 2 – Identify the exception.**
  • Python raises a ZeroDivisionError because division by zero is mathematically undefined and Python cannot produce a numeric result.
  • **Step 3 – Understand what "exception" means.**
  • An exception is an event that disrupts the normal flow of a program during execution (a runtime error). It is different from a syntax error, which is caught before the program runs. When Python raises ZeroDivisionError, execution stops immediately and an error message (traceback) is printed unless the exception is caught.
  • **Step 4 – Other common exceptions to know.**
  • ValueError: raised when a function receives an argument of the correct type but an inappropriate value (e.g., int("abc")).
  • TypeError: raised when an operation is applied to an object of an inappropriate type (e.g., "5" + 5).
  • IndexError: raised when a sequence index is out of range.
  • FileNotFoundError: raised when a file cannot be found.
  • **Answer:** Line 3 raises ZeroDivisionError because y equals 0.

Basic try-except Structure – Guarding Against ZeroDivisionError

  • **Problem:** Rewrite the division script from the previous problem so that it catches the ZeroDivisionError and prints a friendly message instead of crashing.
  • **Step 1 – Recall the basic try-except syntax.**
  • ```
  • try:
  • # code that might raise an exception
  • except ExceptionType:
  • # code that runs if that exception occurs
  • ```
  • Python first executes the try block. If no exception occurs, the except block is skipped. If the specified exception occurs, Python jumps immediately to the except block.
  • **Step 2 – Write the guarded script.**
  • ```
  • x = int(input("Enter numerator: "))
  • y = int(input("Enter denominator: "))
  • try:
  • result = x / y
  • print("Result:", result)
  • except ZeroDivisionError:
  • print("Error: Cannot divide by zero. Please enter a non-zero denominator.")
  • ```
  • **Step 3 – Trace execution for y = 0.**
  • x = 10, y = 0.
  • Python enters the try block and attempts 10 / 0.
  • ZeroDivisionError is raised; Python immediately exits the try block.
  • Python matches the exception to ZeroDivisionError in the except clause.
  • The message "Error: Cannot divide by zero." is printed.
  • The program ends normally (no crash, no traceback).
  • **Step 4 – Trace execution for y = 2.**
  • x = 10, y = 2.
  • 10 / 2 = 5.0 is computed without error.
  • "Result: 5.0" is printed.
  • The except block is never entered.
  • **Key rule:** Place only the code that might fail inside the try block; keep unrelated code outside it to avoid accidentally hiding other errors.

else and finally Clauses – Sensor Data Reader

  • **Problem:** An engineering script opens a file called sensor_data.txt, reads a floating-point temperature value from the first line, and converts it. Use a try-except-else-finally structure so that: the file is always closed, a success message only prints when no error occurs, and file-not-found and value errors are handled separately.
  • **Step 1 – Understand the clause roles.**
  • else: runs only when the try block raised NO exception. Good for code that should execute on success but is not itself risky.
  • finally: runs ALWAYS, whether an exception occurred or not. Ideal for cleanup (closing files, releasing resources).
  • **Step 2 – Write the script.**
  • ```
  • file_handle = None
  • try:
  • file_handle = open("sensor_data.txt", "r")
  • raw = file_handle.read().strip()
  • temperature = float(raw)
  • except FileNotFoundError:
  • print("Error: sensor_data.txt was not found.")
  • except ValueError:
  • print(f"Error: '{raw}' is not a valid floating-point number.")
  • else:
  • print(f"Temperature read successfully: {temperature} °C")
  • finally:
  • if file_handle is not None:
  • file_handle.close()
  • print("File closed.")
  • ```
  • **Step 3 – Trace when file exists and contains "23.7".**
  • open() succeeds; float("23.7") = 23.7 – no exception.
  • else block runs: prints "Temperature read successfully: 23.7 °C".
  • finally block runs: file is closed.
  • **Step 4 – Trace when file does not exist.**
  • open() raises FileNotFoundError.
  • Python jumps to the first except clause; prints the not-found message.
  • else block is SKIPPED (an exception occurred).
  • finally block still runs (file_handle is None so close is skipped safely).
  • **Step 5 – Trace when file contains "ABC".**
  • open() succeeds; raw = "ABC"; float("ABC") raises ValueError.
  • Second except clause runs; prints the invalid-value message.
  • else skipped; finally runs and closes the file.
  • **Answer:** The else clause fires only on clean success; finally fires unconditionally, making it the correct place for resource cleanup.

Accessing Exception Information – Logging Detailed Error Messages

  • **Problem:** A script processes a list of voltage readings stored as strings. For each string, it converts the value to a float and computes power using P = V²/R with R = 50 Ω. If conversion fails, print the specific error message supplied by Python and indicate which reading caused it. Use the list: ["120.5", "115.0", "BAD", "118.3", "110/5"].
  • **Step 1 – Syntax for accessing exception details.**
  • ```
  • except ExceptionType as e:
  • print(e) # prints the error message string
  • print(type(e)) # prints the exception class
  • ```
  • The variable e (any name works) is bound to the exception object, which contains a human-readable description.
  • **Step 2 – Write the processing loop.**
  • ```
  • R = 50
  • readings = ["120.5", "115.0", "BAD", "118.3", "110/5"]
  • for i, reading in enumerate(readings):
  • try:
  • V = float(reading)
  • P = V**2 / R
  • print(f"Reading {i}: V={V} V, P={P:.2f} W")
  • except ValueError as e:
  • print(f"Reading {i}: Conversion error for '{reading}' -> {e}")
  • ```
  • **Step 3 – Trace each iteration.**
  • i=0, "120.5": float("120.5")=120.5; P=120.5²/50=290.405 W. Prints success.
  • i=1, "115.0": float("115.0")=115.0; P=115²/50=264.50 W. Prints success.
  • i=2, "BAD": float("BAD") raises ValueError. e contains: could not convert string to float: 'BAD'. Prints error message.
  • i=3, "118.3": float("118.3")=118.3; P≈280.34 W. Prints success.
  • i=4, "110/5": float("110/5") raises ValueError because the slash makes it invalid as a float literal. Prints error message.
  • **Step 4 – Why "110/5" fails.**
  • float() does not evaluate arithmetic expressions. It only parses numeric strings. "110/5" contains a slash, so it is not a valid float representation.
  • **Key takeaway:** Using as e gives programmers detailed diagnostics useful for logging and debugging in real engineering applications.

Raising Exceptions – Validating Engineering Input

  • **Problem:** Write a function compute_strain(delta_L, L0) that calculates engineering strain as ε = ΔL / L0. The function must raise a ValueError if L0 ≤ 0 (non-physical original length) and raise a TypeError if either argument is not a number. Demonstrate calling the function with valid and invalid inputs.
  • **Step 1 – Syntax for raising exceptions.**
  • ```
  • raise ExceptionType("descriptive message")
  • ```
  • raise intentionally triggers an exception. Use it to enforce preconditions (contracts) that callers must satisfy.
  • **Step 2 – Write the function.**
  • ```
  • def compute_strain(delta_L, L0):
  • # Type check
  • if not isinstance(delta_L, (int, float)):
  • raise TypeError(f"delta_L must be a number, got {type(delta_L).__name__}")
  • if not isinstance(L0, (int, float)):
  • raise TypeError(f"L0 must be a number, got {type(L0).__name__}")
  • # Value check
  • if L0 <= 0:
  • raise ValueError(f"L0 must be positive (original length), got {L0}")
  • return delta_L / L0
  • ```
  • **Step 3 – Test with valid input.**
  • compute_strain(0.005, 2.0)
  • Both values pass isinstance check.
  • L0 = 2.0 > 0, passes value check.
  • Returns 0.005 / 2.0 = 0.0025.
  • **Step 4 – Test with L0 = -1.**
  • compute_strain(0.005, -1)
  • isinstance checks pass (both are numbers).
  • L0 = -1 ≤ 0 → raise ValueError("L0 must be positive, got -1").
  • Execution stops in the function and propagates the exception to the caller.
  • **Step 5 – Test with a string argument.**
  • compute_strain("5mm", 2.0)
  • isinstance("5mm", (int, float)) is False.
  • raise TypeError("delta_L must be a number, got str").
  • **Step 6 – Catching raised exceptions in calling code.**
  • ```
  • try:
  • strain = compute_strain(0.005, 0)
  • except ValueError as e:
  • print(f"Input error: {e}")
  • ```
  • Output: Input error: L0 must be positive (original length), got 0
  • **Best practice:** Raise specific, descriptive exceptions early (fail fast) so bugs are caught at the source rather than producing silent wrong answers downstream.

Practical Error Handling – Robust Engineering Data Pipeline

  • **Problem:** Write a complete Python script that reads a CSV file named beam_loads.csv where each row contains two values: beam_id (string) and load_kN (number). The script must: (1) handle a missing file gracefully, (2) skip rows with non-numeric load values while logging them, (3) raise a ValueError if any valid load is negative, (4) print a summary of successfully processed beams, and (5) always report how many rows were attempted. Use the following simulated file content (create it programmatically in the script for testing): rows = [("B001","150.5"), ("B002","ABC"), ("B003","200.0"), ("B004","-50"), ("B005","175.0")].
  • **Step 1 – Plan the error-handling layers.**
  • Outer layer: FileNotFoundError when opening the file.
  • Middle layer: ValueError from float() for non-numeric strings (skip row, log it).
  • Inner layer: ValueError raised manually for negative loads (skip row, log it).
  • finally layer: always print the row-attempt count.
  • **Step 2 – Write the full script.**
  • ```
  • import csv, os
  • -
  • # --- Create test file ---
  • rows = [("B001","150.5"),("B002","ABC"),("B003","200.0"),
  • ("B004","-50"),("B005","175.0")]
  • with open("beam_loads.csv","w",newline="") as f:
  • writer = csv.writer(f)
  • writer.writerow(["beam_id","load_kN"])
  • writer.writerows(rows)
  • -
  • # --- Process file ---
  • processed = []
  • skipped = []
  • attempted = 0
  • -
  • try:
  • file = open("beam_loads.csv","r",newline="")
  • except FileNotFoundError:
  • print("CRITICAL: beam_loads.csv not found. Pipeline aborted.")
  • file = None
  • -
  • if file is not None:
  • try:
  • reader = csv.DictReader(file)
  • for row in reader:
  • attempted += 1
  • beam_id = row["beam_id"]
  • raw_load = row["load_kN"]
  • try:
  • load = float(raw_load)
  • if load < 0:
  • raise ValueError(
  • f"Negative load {load} kN is physically invalid.")
  • processed.append((beam_id, load))
  • except ValueError as e:
  • skipped.append((beam_id, raw_load, str(e)))
  • finally:
  • file.close()
  • print(f"\nRows attempted: {attempted}")
  • -
  • print("\n--- Successfully Processed Beams ---")
  • for bid, load in processed:
  • print(f" {bid}: {load} kN")
  • -
  • print("\n--- Skipped Rows ---")
  • for bid, raw, reason in skipped:
  • print(f" {bid} (raw='{raw}'): {reason}")
  • ```
  • **Step 3 – Trace row by row.**
  • B001, "150.5": float("150.5")=150.5; 150.5 ≥ 0 → added to processed.
  • B002, "ABC": float("ABC") raises ValueError("could not convert..."); caught → added to skipped.
  • B003, "200.0": float OK; 200.0 ≥ 0 → processed.
  • B004, "-50": float("-50")= -50.0; -50 < 0 → manual raise ValueError("Negative load..."); caught → skipped.
  • B005, "175.0": float OK; processed.
  • **Step 4 – Expected output.**
  • Rows attempted: 4 (header row excluded by DictReader)
  • Processed: B001 150.5 kN, B003 200.0 kN, B005 175.0 kN.
  • Skipped: B002 (non-numeric), B004 (negative load).
  • **Step 5 – Key design principles demonstrated.**
  • Nest try-except blocks at the appropriate granularity: file-open errors are fatal; per-row errors are recoverable.
  • Use finally for guaranteed cleanup (file.close()) regardless of what happens inside the loop.
  • Use raise to enforce domain rules (no negative loads) with the same mechanism as built-in errors, keeping error handling uniform.
  • Log enough information (beam_id, raw value, reason) so engineers can correct the source data.
